# 0017 — リポジトリの .ps1 は pwsh 7 で実行する（Windows PowerShell 5.1 は文字化けでパースエラー）

## 症状

`teardown.ps1` 等を実行すると、**スクリプト本文の構文エラー**で何もせず落ちる。

```
発生場所 teardown.ps1:73 文字:69
+ ... ("  NAT スイッチ : {0} (+ {0}-NAT)" ...   ← 文字化けした行
式またはステートメントのトークン '-NAT' を使用できません。
文字列に終端記号 " がありません。
```

エラー箇所のソースが化けている（例: `スイッチ` が別の字に）。コード自体は正しいのに落ちるのが特徴。

## 原因

本リポジトリの **.ps1 は全て UTF-8（BOM なし）＋日本語コメント/文字列入り**。
**Windows PowerShell 5.1 は、BOM の無いスクリプトを現在の ANSI コードページ（日本語環境＝cp932）で読む**。
日本語のバイト列が cp932 として誤解釈され、たまたま 2 バイト目が `"` や `)` `\` 等の構文上重要な文字に
化けると、文字列終端やカッコ対応が崩れてパースエラーになる。`bootstrap.ps1` は化けても構文を壊さない
位置だっただけで、本質は同じ。**ファイルは壊れていない。読み方（実行系）が違う。**

## 対策

**このリポジトリのスクリプトは PowerShell 7（`pwsh`）で実行する。** pwsh は .ps1 を既定で UTF-8 として
読むため、BOM なしでも正しくパースされる（実機確認: 全 .ps1 を pwsh7 でパース→構文エラー 0）。

```powershell
pwsh -NoProfile -File .\teardown.ps1 -Force          # ○ 正しい
powershell -File .\teardown.ps1                       # × 5.1。cp932 誤読で落ちる
```

ホストには `C:\Program Files\PowerShell\7\pwsh.exe`（7.5.x）が導入済み。
SSH 経由で叩く場合も `ssh nestedhyperv "pwsh -NoProfile -EncodedCommand <UTF16LE-base64>"` を使う。

（恒久的にどちらでも動かしたい場合の代替: 全 .ps1 を UTF-8 with BOM で保存し直せば 5.1 でも読める。
ただし本プロジェクトは pwsh7 前提で統一する方針なので、実行系を pwsh に固定するのが正。）

## 教訓 / 汎用ノウハウ

- **「コードは正しいのに構文エラー」はエンコーディング × 実行系の不一致を疑う。** 特に日本語入り .ps1 ×
  Windows PowerShell 5.1（cp932）は定番の罠。エラー行のソースが化けていたら確定。
- **同一マシンに 5.1 と 7 が共存する。** 既定の `powershell` は 5.1。本リポジトリは必ず `pwsh`。
